Task Definitions
Task 1 Gradle Multi-Module Project
What it is
Convert the current IntelliJ-only vr_physics-master project into a proper Gradle multi-module build with three subprojects: :shared, :sim, and :robot-code. The build lives at the repo root (FTC_Robot_Sim/) alongside the existing Android project.
What it is NOT
Not an Android build — this is pure Java 17 / JavaFX. The Android project (Darlington_in_FTC copy/) stays untouched and continues to build via Android Studio separately. We are building the simulator side.
Why this comes first: Every other task (interfaces, sim implementations, robot-code injection) requires a working Gradle build with proper module boundaries. Without it, there is no clean place to put the shared interfaces and no way to verify that :robot-code genuinely cannot import from :sim.
Task 2 IMotor, IServo, IIMU, IHardwareMap Interfaces
What they are
Four platform-neutral Java interfaces that live in :shared and act as the seam between robot code and the simulator. Robot code calls through these interfaces; the sim provides implementations. The same interfaces will later wrap the real FTC SDK on Android — so they must expose exactly the methods Darlington_in_FTC actually uses, nothing more.
What they are NOT
Not a full re-implementation of the FTC SDK. We are not wrapping every method of DcMotorEx or BNO055IMU — only the subset that MecanumDrive.java, Shooters.java, and Auto.java actually call. Anything unused stays out of the interface — it can be added later without breaking anything.
Why these four specifically: Motor (drive + mechanism actuators), Servo (intake / shooter angle), IMU (heading for MecanumDrive auto), HardwareMap (device registry). These four cover 100% of the hardware calls in the Darlington codebase. Freezing them at Sync 1 lets Track A and Track B diverge safely.

Dependency rule: :shared has no dependencies. :sim depends on :shared + JavaFX + ODE4J. :robot-code depends on :shared only. This is enforced at the Gradle level — any attempt to import from :sim inside :robot-code will fail the build.
Interface Method Specifications
Methods listed are derived from auditing MecanumDrive.java, Shooters.java, and Auto.java in the Darlington codebase. Only used methods appear. Enums needed by the methods are nested inside the interface.
IMotor
void setPower(double power)
double getPower()
void setMode(RunMode mode)
RunMode getMode()
int getCurrentPosition()
void setTargetPosition(int ticks)
boolean isBusy()
void setVelocity(double ticksPerSec)
void setZeroPowerBehavior(ZeroPowerBehavior b)
void setDirection(Direction d)
Nested enums: RunMode {RUN_TO_POSITION, RUN_USING_ENCODER,
RUN_WITHOUT_ENCODER, STOP_AND_RESET_ENCODER},
ZeroPowerBehavior {BRAKE, FLOAT}, Direction {FORWARD, REVERSE}
IServo
void setPosition(double position)
double getPosition()
void scaleRange(double min, double max)
void setDirection(Direction d)
position is always [0.0, 1.0] in interface contract.
Nested enum: Direction {FORWARD, REVERSE}
IIMU
double getHeading(AngleUnit unit)
double getRoll(AngleUnit unit)
double getPitch(AngleUnit unit)
void resetHeading()
AngleUnit enum: DEGREES, RADIANS (same as FTC SDK).
Heading is robot yaw in the field frame, zero at init.
IHardwareMap
<T> T get(Class<T> type, String name)
IMotor getMotor(String name)
IServo getServo(String name)
IIMU getIMU(String name)
Convenience getters keep robot code readable.
Generic get() is for future extensibility only.
How We Will Work — Ordered Steps
1
Audit Darlington hardware calls
Before writing any interface, grep Darlington_in_FTC copy/TeamCode/ for every hardware method call in MecanumDrive.java, Shooters.java, and Auto.java. Build the exact method list — nothing assumed. Any method not found in the grep stays out of the interface.
2
Create root Gradle project
Write FTC_Robot_Sim/settings.gradle and build.gradle. Root build sets Java 17 toolchain, adds shared Maven repos (ODE4J, JavaFX). settings.gradle includes :shared, :sim, :robot-code. Verify ./gradlew projects lists all three.
3
Scaffold :shared module + write interfaces
Create shared/build.gradle (java-library, no runtime deps). Write all four interfaces in shared/src/main/java/org/darbots/sim/api/ using the audited method list from Step 1. Nested enums go inside their interface file. Verify ./gradlew :shared:compileJava passes clean.
4
Scaffold :sim module — wire vr_physics source
Create sim/build.gradle with dependencies: project(':shared'), ODE4J jar from vr_physics-master/lib/ (local file dep), JavaFX via org.openjfx plugin. Point sourceSets.main.java.srcDirs to ../vr_physics-master/Controller/src — no copy, no symlink. Verify ./gradlew :sim:compileJava passes.
5
Scaffold :robot-code module
Create robot-code/build.gradle with exactly one project dependency: project(':shared'). Add a placeholder src/main/java/.gitkeep. Verify the build compiles and that adding any :sim import fails as expected (dependency boundary test).
6
Wire :sim run task → JavaFX window
Add a Gradle application plugin run task in sim/build.gradle pointing to the existing vr_physics main class. Run ./gradlew :sim:run and confirm the JavaFX simulator window opens. This is the visual proof-of-life that the Gradle wiring is complete.
7
Add stub implementations in :sim for all four interfaces
Write SimMotor, SimServo, SimIMU, SimHardwareMap in :sim — each implements the matching interface from :shared. These are stubs for now (no physics). Verify :sim still compiles. These stubs will be fleshed out in E2/E1.
✓
Commit + tag Sync 1 checkpoint
Commit everything. Verify the full build: ./gradlew build green, ./gradlew :sim:run opens window. Update progress.html: E0 → 100%, Track A + B → unblocked. This commit is the gate that unlocks E2 and E1.

Risk to watch during this work
The biggest risk is interface scope creep — adding methods "just in case" that aren't actually used in the Darlington code. Every extra method is something both the SimMotor stub and the future real-robot adapter must implement. Stick to the audit: if it's not called in TeamCode, it stays out. A too-thin interface can always grow; a too-fat one forces work across both tracks.
